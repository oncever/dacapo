Known Problems

  • The 'chart' benchmark will not run on VMs which use GNU Classpath 0.91 or earlier.

  • The 'eclipse' benchmark requires a regular, Sun-like jre in order to be able to perform its build tasks (it needs to be able to compile against standard libraries such as 'rt.jar'). Some JVMs do not comply with eclipse's expectations. A simple work-around is to specify a jre for eclipse to compile against. If the jvm used to run the benchmarks does not comply with eclipse's expectations, an error message will be produced with an explanation of the suggested work around (setting -Declipse.java.home=xxx on the command line to point to an orthodox jre, whose libraries will be used).

  • The 'jython' benchmark has been reported to fail bytecode verification on an internal product JVM. We have not yet confirmed whether this is a problem with jython or just that JVM. The jython interpreter dynamically generates bytecodes, and apparently it is these that are failing verification.

  • The benchmarks run with the Sun jdk 1.4 and 1.5, but do not build with a 1.5 javac. This is a limitation of some of the benchmarks themselves.

  • The jar file contains the compiled classes for the benchmark programs, plus the minimum library jar file contents required to get the benchmarks to run. If the system class libraries for your VM are missing classes which are distributed as library jar files with the benchmark, please let us know so we can amend the build procedure. After building with ant, the source tree for each benchmark is left in the build_src subdirectory, and comparing the library jar files distributed with the benchmark and the dacapo build.xml might provide the answer.

  • There are some issues specific to Jikes RVM.
    • By default, JikesRVM does not build classpath with support for AWT, which chart uses. Jikes RVM can be configured to use AWT, by editing rvm/bin/jBuildClasspathJar, in particular, ensuring that the --disable-gtk-peer argument is not set when classpath is built (ie you want the gtk-peer enabled).
    • We found that a number of benchmarks, most notably 'eclipse', exposed brittleness in Jikes RVM, such that some configurations would not relibaly run or would only run at some heap sizes. Status as at Jikes RVM version 2.4.5:
      • MarkCompact will not run for 'eclipse' or 'jython', and will only work in large heaps for 'luindex'.
      • GenMS will not run for 'eclipse', and will only work in large heaps for 'luindex'.
      • GenCopy will only work in large heaps for 'luindex'.
    • Version beta041020 of the DaCapo suite did not include sufficient classes for Jikes RVM to run 'xalan'.  This is fixed in subsequent releases.
