Known Problems

  • The 'chart' benchmark will not run on VMs which use GNU Classpath 0.91 or earlier.

  • The 'eclipse' benchmark requires a regular jre in order to be able to perform its build tasks (it does needs to be able to compile against standard libraries). Some JVMs do not comply with eclipse's expectations. A simple work-around is to specify a jre for eclipse to compile against. If the jvm used to run the benchmarks does not comply with eclipse's expectations, an error message will be produced with an explanation of the suggested work around (setting -Declipse.java.home=xxx on the command line).

  • The benchmarks run with the Sun jdk 1.4 and 1.5, but do not build with 1.5. This is a limitation of some of the benchmarks themselves.

  • The jar file contains the compiled classes for the benchmark programs, plus the minimum library jar file contents required to get the benchmarks to run.

    If the system class libraries for your VM are missing classes distributed as library jar files with the benchmark, please let us know so we can amend the build procedure. After building with ant, the source tree for each benchmark is left in the build_src subdirectory, and comparing the library jar files distributed with the benchmark and the dacapo build.xml might provide the answer.

  • There are some issues with JikesRVM.
    • By default, JikesRVM does not build classpath with support for AWT, which batik and chart use. Jikes can be configured to use AWT, by editing rvm/bin/jBuildClasspathJar, in particular, ensuring that the --disable-gtk-peer argument is not set when classpath is built (ie you want the gtk-peer enabled).

    • Version beta041020 did not include sufficient classes for Jikes RVM to run xalan.  This is fixed in subsequent releases.
